## 10. 답변 검토 후 추천안

> 작성일: 2026-07-13
> 아래 내용은 기존 답변을 바꾸지 않고 덧붙인 제안입니다. `추천`은 기본 설계안이고, `확인 필요`는 구현 전에 사용자가 최종 결정해야 하는 항목입니다. 완료일 `07/31`은 **2026-07-31(KST)**로 해석했습니다.

### 프로젝트를 한 문장으로 정리한 추천 정의

**MyWiki는 가입 회원마다 하나의 비공개 개인 위키를 제공하고, 소유자가 선택한 문서만 다른 회원에게 열람자 또는 편집자 권한으로 공유하는 Markdown 기반 지식 관리 서비스입니다.**

일반적인 단일 공용 위키보다 여러 사용자의 데이터가 분리되는 **다중 사용자 개인 지식 공간**에 가깝습니다. 따라서 모든 데이터 조회에서 현재 사용자가 해당 위키나 문서에 접근할 수 있는지 확인하는 것이 가장 중요한 설계 원칙입니다.

### 먼저 확인해야 할 결정

- [x] **회원 간 연결의 의미(Q51):** MVP에서는 친구·팔로우 관계가 아니라, 기존 회원을 검색하여 특정 문서의 `열람자` 또는 `편집자`로 초대하는 기능으로 정의하는 것을 추천합니다.
- [x] **공유 단위(Q4):** MVP에서는 문서 단위 공유만 구현하고, 위키 전체 공유는 후속 기능으로 두는 것을 추천합니다.
- [x] **로그인 범위(Q6):** 7월 31일까지는 이메일·비밀번호만 구현하고 Google·GitHub·Kakao 로그인은 후속 기능으로 미루는 것을 추천합니다.
- [x] **공개 범위(Q7):** MVP는 `비공개`와 `지정 회원 공유`만 지원하고, 비회원 공개와 검색 엔진 노출은 후속 기능으로 두는 것을 추천합니다.
- [x] **삭제 정책(Q28):** 즉시 완전 삭제 대신 30일 휴지통 후 영구 삭제로 바꾸는 것을 추천합니다.
- [x] **첨부 범위(Q18):** MVP는 JPG·PNG·WebP 이미지 업로드만 지원하고, PDF 등 일반 파일은 후속 기능으로 두는 것을 추천합니다.
- [x] **관리자 권한(Q20~21):** 관리자는 계정과 서비스 상태만 관리하고, 사용자의 비공개 문서 본문은 기본적으로 열람할 수 없게 하는 것을 추천합니다.
- [x] **일정(Q12):** 아래의 MVP 범위를 2026-07-31 완료 목표로 사용하는 것을 추천합니다.

### 답변 사이에서 발견한 충돌

1. **검색(Q3, Q30):** 첫 버전 기능에서는 검색이 선택되지 않았지만, Q30에는 제목·본문·태그 검색이 필요하다고 적혀 있습니다. 위키의 핵심 탐색 수단이므로 권한이 있는 문서만 대상으로 하는 기본 검색을 MVP에 포함하는 것을 추천합니다.
2. **회원 권한(Q4, Q21):** “허가받은 사람만 열람·수정”과 “회원은 열람·작성·수정·삭제”가 충돌합니다. 회원이라는 이유만으로 다른 회원의 문서에 접근할 수 없도록 권한을 분리해야 합니다.
3. **이메일 기능(Q22, Q45):** 이메일 인증과 비밀번호 재설정에는 트랜잭션 이메일 발송 서비스가 필요하므로 “외부 연동 없음”과 충돌합니다. 이메일 발송 서비스 하나는 필수 연동으로 보는 것이 좋습니다.
4. **문서 이력과 삭제(Q24~25, Q28):** 모든 버전을 보관하면서 즉시 완전 삭제하면 복구 요구와 충돌합니다. 휴지통 보존 기간과 백업에서 최종 삭제되는 시점을 함께 정해야 합니다.
5. **공개 상태(Q7, Q13, Q20):** 현재는 비공개이며 비회원 기능도 없으므로, MVP의 공개 상태는 `비공개`와 `공유됨`만 있으면 됩니다. `전체 공개`는 HTTPS·운영 정책·신고 대응을 준비한 뒤 추가하는 편이 안전합니다.

### 질문별 세부 추천

#### 목표, 기능, 기술(Q1~12)

- **Q1~2:** “개인 지식 관리와 선택적 공유”를 핵심 가치로 두고, 회원당 위키 1개로 시작하는 것이 좋습니다. 한 회원이 여러 위키를 갖는 기능은 데이터 모델만 확장 가능하게 설계하고 UI에서는 막아 둡니다.
- **Q3·Q30:** 검색을 MVP 기능으로 추가합니다. 검색 결과에는 사용자가 소유하거나 공유받은 문서만 나와야 합니다.
- **Q4·Q20~21:** 플랫폼 역할과 문서 권한을 분리합니다.
  - 일반 회원: 자기 위키와 자기 문서를 관리
  - 문서 소유자: 열람·수정·공유·삭제·복구
  - 문서 편집자: 열람·수정
  - 문서 열람자: 열람만 가능
  - 관리자: 계정 비활성화 등 서비스 관리만 담당하고 비공개 본문은 기본 열람 불가
- **Q5:** Markdown 원문 안의 HTML은 MVP에서 비활성화하고, 렌더링된 HTML은 허용 목록 기반으로 정제합니다. 사용자 본문에 Jinja의 `safe`를 직접 적용하지 않습니다.
- **Q6·Q22:** 이메일·비밀번호부터 완성합니다. 비밀번호는 Argon2id로 해시하고, 인증·재설정 토큰은 짧은 만료 시간과 1회 사용, 요청 횟수 제한을 적용합니다. 소셜 로그인은 먼저 한 공급자만 후속 추가하는 것이 좋습니다.
- **Q7:** 모든 위키와 문서는 기본 비공개로 생성합니다. 나중에 공개 기능을 추가하더라도 문서별로 소유자가 명시적으로 켜게 합니다.
- **Q8~9:** PC 중심의 2단 레이아웃을 사용하되, 작은 화면에서는 사이드바가 접히는 반응형 구조로 만듭니다. Notion의 여백과 편집 경험, 위키의 링크·이력·탐색 구조를 조합하는 방향이 적합합니다.
- **Q10:** 일정과 기존 방향을 고려하면 다음 조합을 추천합니다.
  - Flask 3.1 계열, Jinja2, Bootstrap 5.3 계열, 소량의 JavaScript
  - Flask 애플리케이션 팩토리와 기능별 Blueprint
  - SQLAlchemy 2 계열과 Alembic 데이터베이스 마이그레이션
  - PostgreSQL, pytest, Ruff
  - Bootstrap 위에 전용 CSS 변수와 컴포넌트 스타일을 추가해 기본 Bootstrap처럼 보이지 않도록 커스텀 테마 구성
- **Q11:** 배포 업체는 예산과 예상 사용량을 확인한 뒤 정합니다. 우선은 `Linux 애플리케이션 컨테이너 + 관리형 PostgreSQL + 비공개 S3 호환 객체 저장소 + 기본 HTTPS`를 제공하는 PaaS를 기준으로 공급자에 종속되지 않게 구성하는 것을 추천합니다.
- **Q12:** 현재 날짜 기준 남은 기간이 짧으므로, 위의 MVP를 먼저 완성하고 제외 항목은 7월 31일 이후 단계로 나누는 것이 현실적입니다.

#### 콘텐츠와 문서 구조(Q13~19)

- **Q13:** 적어 둔 필드에 `wiki_id`, `owner_id`, `현재 revision 번호`, `삭제 상태`, `공유 범위`를 추가합니다. 태그와 즐겨찾기는 별도 관계 테이블로 관리합니다.
- **Q14:** 태그·최근 문서·즐겨찾기를 유지하고, 계층형 카테고리와 폴더 트리는 제외하는 것을 추천합니다.
- **Q15~16:** `/w/{사용자명}/docs/{문서 UUID}` 형식을 추천합니다. 문서 주소는 UUID에 고정되어 제목이 바뀌어도 유지됩니다. UUID가 있어도 보안이 보장되는 것은 아니므로 요청할 때마다 권한을 검사해야 합니다.
- **Q17:** MVP에서는 문서 선택기가 고정 문서 ID를 사용하는 Markdown 링크를 삽입하게 하는 방식을 추천합니다. 제목 변경에도 링크가 깨지지 않습니다. `[[문서 제목]]` 자동 완성과 역링크는 후속 기능으로 두는 편이 안전합니다.
- **Q18:** 파일 본문을 별도 데이터베이스에 넣는 방식은 추천하지 않습니다. 운영에서는 비공개 객체 저장소에 파일을 두고 PostgreSQL에는 소유자·저장 키·원래 이름·MIME 형식·크기 같은 메타데이터만 저장합니다. MVP 이미지는 파일당 5MB로 시작하고 SVG와 실행 파일은 허용하지 않는 것을 추천합니다.
- **Q19:** 마이그레이션 대상이 없으므로 샘플 데이터 가져오기 기능은 만들지 않아도 됩니다.

#### 사용자, 권한, 편집(Q20~29)

- **Q23:** 필수 정보는 내부 사용자 ID, 고유 사용자명, 표시 이름, 이메일, 이메일 인증 시각, 가입일입니다. 아바타와 소개는 선택 사항으로 두고 이메일은 외부에 공개하지 않습니다.
- **Q24~25:** 문서를 저장할 때마다 전체 Markdown 스냅샷을 불변 revision으로 남기는 방식이 MVP에 단순하고 안전합니다. 이전 버전 복구도 과거 이력을 덮어쓰지 않고 새로운 revision을 생성해야 합니다. 공유 전 revision에 삭제된 민감정보가 있을 수 있으므로 MVP에서는 이력 열람·비교·복구를 문서 소유자에게만 허용하는 것을 추천합니다.
- **Q26:** 문서 잠금 대신 낙관적 잠금을 사용합니다. 편집을 시작한 revision과 저장 시점의 revision이 다르면 저장을 중단하고 충돌 내용을 보여 줍니다.
- **Q27:** 편집 요약은 선택 입력으로 유지하되, 비어 있으면 `문서 생성`, `본문 수정`, `제목 변경`, `이전 버전 복구` 같은 기본 사유를 자동 기록합니다.
- **Q28:** 기본 삭제는 30일 휴지통으로 하고, 영구 삭제는 소유자만 재확인 후 실행하도록 추천합니다. 영구 삭제 뒤에도 백업 사본은 정해진 보존 기간이 지나야 사라진다는 점을 운영 정책에 명시해야 합니다.
- **Q29:** “제한적 필요”의 범위는 MVP에서 제외하는 것으로 정리하는 것을 추천합니다. 이후 가장 먼저 추가한다면 문서 공유 초대와 권한 변경에 대한 알림부터 시작합니다.

#### 검색과 화면(Q30~37)

- **Q30:** PostgreSQL의 단순 부분 검색으로 시작하고, 문서가 늘면 `pg_trgm` 인덱스를 추가합니다. 한글 형태소 분석·초성 검색은 별도 검색 엔진이나 확장이 필요할 수 있으므로 후속 범위로 둡니다.
- **Q31:** 대문은 `통합 검색`, `새 문서`, `최근 문서`, `공유받은 문서`, `즐겨찾기` 순으로 구성하는 것을 추천합니다.
- **Q32:** 사이드바에는 `홈 / 모든 문서 / 최근 문서 / 즐겨찾기 / 태그 / 공유받은 문서 / 휴지통 / 설정`, 상단에는 현재 문서 경로·검색·공유 버튼·프로필·테마 전환을 둡니다.
- **Q33:** 초기에는 별도 로고 제작보다 중립적인 회색 계열에 강조색 하나를 정하고 색상·간격·모서리·그림자를 CSS 변수로 관리하는 디자인 시스템을 추천합니다.
- **Q34:** 운영체제 설정을 따르는 `자동`을 기본으로 하고 `라이트 / 다크 / 자동` 선택을 사용자 설정에 저장합니다. Bootstrap의 `data-bs-theme`와 CSS 변수를 활용할 수 있습니다.
- **Q35:** UI는 한국어만 제공하되, 화면 문자열을 템플릿에 흩어 놓지 않아 나중에 다국어 처리하기 쉽게 구성합니다.
- **Q36:** 키보드만으로 모든 기능을 사용할 수 있게 하고 WCAG 2.2 AA를 기본 목표로 둡니다. 본문 바로가기, 항상 보이는 포커스, 올바른 폼 레이블, 모달 포커스 관리, 키보드 트랩 방지가 필요합니다. 단축키는 `/` 검색, `Ctrl+S` 저장, `Esc` 닫기 정도만 제공하고 입력 중에는 오작동하지 않게 합니다.
- **Q37:** 우선 최신 Chrome과 Edge를 주요 지원 대상으로 두고, 최신 Firefox와 Safari에서도 핵심 열람·편집 흐름이 동작하는지 확인합니다.

#### 데이터, 운영, 보안(Q38~49)

- **Q38:** 요구가 확정될 때까지 `회원 100명 이하, 동시 사용자 20명 이하, 전체 문서 10,000개 이하, 전체 첨부 10GB 이하`를 초기 설계·테스트 가정으로 두는 것을 추천합니다. 이는 제한값이 아니라 첫 용량 계획 기준입니다.
- **Q39:** 운영과 통합 테스트는 PostgreSQL을 추천합니다. SQLite는 간단한 단위 테스트나 임시 로컬 실행에만 사용합니다. 공유 권한, 검색, revision, 동시 편집을 운영 DB와 같은 조건에서 검증하는 편이 안전합니다.
- **Q40:** `매일 자동 백업, 30일 보관, 월 1회 복구 시험, RPO 24시간, RTO 4시간`을 초기 기준으로 추천합니다. 데이터베이스와 첨부 저장소는 같은 시점으로 복구할 수 있어야 하며, 마이그레이션 직전에는 별도 백업을 남깁니다.
- **Q41:** 이메일, 비밀번호 해시, 비공개 문서, 첨부, 공유 관계를 민감정보로 취급합니다. HTTPS, 저장소 암호화, 최소 권한, CSRF 방어, 안전한 세션 쿠키, 로그인·재설정·초대·업로드 속도 제한, 비밀 값의 환경 분리가 필요합니다.
- **Q42:** 감사 로그는 필요합니다. 로그인 결과, 이메일·비밀번호 변경, 공유 권한 변경, 관리자 행위, 삭제·복구·영구 삭제를 남깁니다. 행위자·대상·결과·UTC 시각·요청 ID는 기록하되 비밀번호, 토큰, 문서 본문은 기록하지 않습니다. 초기 보존 기간은 180일을 추천합니다.
- **Q43:** 로컬 개발에는 도메인과 인증서가 없어도 되지만 외부 공개 전에는 HTTPS를 필수 조건으로 둡니다. 초기에는 배포 업체의 기본 HTTPS 주소를 사용하고 커스텀 도메인은 나중에 연결해도 됩니다.
- **Q44:** MVP에는 구조화된 서버 로그와 `/health/live`, `/health/ready` 상태 확인을 포함합니다. 운영 시 오류 추적과 5xx·응답 지연·DB 상태·용량·백업 실패 알림을 추가하며, 로그와 오류 보고에는 문서 본문·이메일·토큰을 보내지 않습니다.
- **Q45:** 이메일 인증과 비밀번호 재설정을 위한 트랜잭션 이메일 서비스가 첫 외부 연동입니다. 개발 환경에서는 실제 발송 대신 메일 내용을 로컬에서 확인할 수 있게 분리합니다.
- **Q46:** 외부 API는 MVP에서 만들지 않습니다. 다만 인증·문서·권한·검색 로직을 화면 라우트와 분리한 서비스 계층에 두어 나중에 API를 추가할 수 있게 합니다.
- **Q47:** Windows 개발은 지원하되 운영은 Linux를 추천합니다. 현재 개발 PC에서 확인된 Python은 3.14.6이므로 의존성 호환성을 검증한 뒤 한 minor 버전으로 고정합니다. `.venv`, UTF-8, `pathlib`, 환경별 설정 파일, 환경 변수 기반 비밀 관리를 사용합니다. Flask 개발 서버와 디버거는 운영에서 사용하지 않습니다.
- **Q48:** 자동 테스트와 CI는 필요합니다. 인증, 사용자 A/B 데이터 격리, 공유 권한 부여·회수, 문서 CRUD·검색·revision 복구, CSRF, Markdown XSS, 업로드, 충돌 처리를 최소 테스트 범위로 둡니다.
- **Q49:** `main` 브랜치와 짧은 기능 브랜치, Ruff 포맷·검사, pytest 통과 후 병합, 데이터베이스 스키마 변경 시 Alembic migration 포함을 최소 규칙으로 추천합니다. 커밋은 `feat:`, `fix:`, `docs:`, `test:`, `chore:` 접두사를 사용하면 이력을 읽기 쉽습니다.

- **Q51:** 다음 문장을 검증 가능한 성공 기준으로 추천합니다.
  **“서로 다른 두 회원의 위키가 기본적으로 완전히 분리되고, 문서 소유자는 선택한 문서만 다른 회원에게 열람 또는 편집 권한으로 공유·회수할 수 있다. 회원은 권한이 있는 문서만 검색할 수 있고, 변경 이력을 비교하여 이전 버전을 복구할 수 있다.”**

### 추천 데이터 모델 초안

사용자마다 별도 데이터베이스를 만드는 대신, **하나의 Flask 모듈식 애플리케이션과 하나의 PostgreSQL 공유 스키마**를 사용하고 모든 위키 소유 데이터에 `wiki_id`를 두는 방식을 추천합니다. 이 구조가 문서 공유·검색·운영·백업을 가장 단순하게 유지합니다.

- `users`: 계정, 프로필, 인증 상태
- `wikis`: 소유자와 개인 위키 설정
- `documents`: 현재 제목·본문·상태와 고정 공개 ID
- `document_revisions`: 문서별 불변 변경 이력
- `document_permissions`: 문서와 공유받은 회원의 `viewer/editor` 권한
- `tags`, `document_tags`: 위키별 태그와 문서 연결
- `favorites`: 회원별 즐겨찾기
- `document_links`: 원본·대상 문서의 고정 ID 연결
- `attachments`: 객체 저장소의 파일 메타데이터
- `email_tokens`: 인증·재설정용 단기 토큰
- `audit_logs`: 공유·삭제·관리자 등 보안 관련 행위

관계의 핵심은 `사용자 → 개인 위키 → 문서 → revision`이며, 다른 사용자의 접근은 반드시 `document_permissions`를 통해서만 허용합니다. 문서 본문뿐 아니라 검색 결과·미리보기·이력·첨부 다운로드에서도 같은 권한 정책을 사용해야 합니다.
